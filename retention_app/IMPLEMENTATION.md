# Implementation Reference

Technical description of the Retention App's architecture, data model, and key design decisions.

---

## Architecture Overview

The application is a single-process FastAPI server with an in-process APScheduler background job. All persistence is handled by a SQLite database accessed via SQLAlchemy ORM.

```
Browser
  │
  ▼
FastAPI (app/main.py)
  ├── Jinja2 HTML templates (app/ui/templates/)
  ├── Content ingestion pipeline (app/ingest/, app/services/content_service.py)
  │     └── LLM question generation (app/llm/)
  ├── Quiz & review endpoints (app/services/quiz_service.py, review_service.py)
  └── APScheduler background job (app/scheduling/scheduler.py)
        └── System / email notifications (app/scheduling/notifications.py)

SQLite (app.db) ← SQLAlchemy ORM (app/db/)
```

## Content Ingestion

### Supported sources

| Source | Detection |
|---|---|
| YouTube video | URL contains `youtube.com` or `youtu.be` |
| Podcast RSS feed | URL ends in `.xml` or content-type is `application/rss+xml` |
| PDF (local file) | Uploaded via multipart form; stored to a temp file |
| PDF (web-hosted) | URL ends in `.pdf` |
| Webpage | All other URLs (trafilatura + readability extraction) |

### Pipeline

1. **Download / extract** (`app/ingest/router.py` dispatches to the appropriate handler)
2. **Transcribe** — audio sources are decoded with ffmpeg and transcribed by OpenAI Whisper
3. **OCR** — video frames are sampled and processed with Tesseract (`app/processing/ocr.py`)
4. **Reconcile** — raw Whisper transcript and OCR tokens are reconciled by an LLM call to correct proper nouns and technical terms (`app/llm/reconcile.py`)
5. **Generate questions** — `create_question_set()` in `quiz_service.py` calls `generate_questions()` which sends the cleaned text to OpenRouter and validates the response against `FreeQuestionSetOutput`
6. **Persist** — `ContentText`, `QuestionSet`, `Question`, and `ScheduleState` records are committed; content status advances to `ready`

Ingestion runs asynchronously after the HTTP response is returned; status can be polled via the content detail page.

### Artifact logging

A `DebugLogger` (`app/llm/debug_logger.py`) writes intermediate outputs (raw transcript, reconciled transcript, OCR corpus, LLM prompts and responses) to `artifacts/content_{id}/` for inspection.

## Question Generation

Questions are generated by `full_text_question_prompt()` (`app/llm/prompts.py`), which instructs the LLM to produce exactly `QUESTIONS_PER_BLOOM_LEVEL` questions at each of the six Bloom's Revised Taxonomy levels:

| Level | Focus |
|---|---|
| remember | Recall of specific facts |
| understand | Paraphrase or explain concepts |
| apply | Use knowledge in a new context |
| analyze | Break down or compare |
| evaluate | Justify or critique |
| create | Synthesise or design |

The LLM response is validated by Pydantic against `FreeQuestionSetOutput` (list of `FreeQuestionOutput`, each with `bloom_level`, `prompt`, `expected_answer`, and `key_points`). Question types are stored as the Bloom level string in the `questions.question_type` column.

---

## Quiz Display

The `GET /quiz/{content_id}/scheduled` endpoint loads the latest `QuestionSet` and passes questions through `_split_questions()` which partitions them by Bloom level relative to `SCHEDULED_BLOOM_CEILING` (default: `understand`):

- **Required** — questions at or below the ceiling (shown in the main numbered list)
- **Optional challenges** — questions above the ceiling (shown in an expanded section below)

Both groups include expected-answer reveal toggles. All questions are visible on page load.

## Scheduling

Strategy is implemented in `app/scheduling/strategy_v1.py`. After each scheduled quiz attempt, the user provides a comfort rating (1–5). The rating is scaled to a score (2–10) and drives the next interval:

| Interval steps | Days until next review |
|---|---|
| 1, 2, 3 | +1 day |
| 4 | +3 days |
| 5 | +5 days |
| 6, 7 | +7 days |
| 8 | +14 days |

Score modifiers:

- Score ≥ 8 (comfort 4–5): next interval extended by 2 days (not applied on the first three attempts)
- Score ≤ 5 (comfort 1–2): step index resets to 0; question set is regenerated
- 5 < score < 8 (comfort 3): no change

After the final step, the `ScheduleState` is marked `is_terminated = True` and the content becomes available for on-demand practice only.

## Notifications

`ReminderScheduler` runs a background job every 5 minutes (`check_due_items`). For each `ScheduleState` with `next_due_at` in the past and no notification sent in the last 12 hours, `_send_notifications()` dispatches:

- **System notification** — via plyer; on Windows, falls back to a PowerShell `NotifyIcon` balloon tip if plyer raises an exception
- **Email reminder** — via `smtplib` STARTTLS; disabled by default

Notification records are written to the `notifications` table with `pending` / `sent` / `failed` status.

---

## User Sessions

Identity is established by submitting a username on the login page. A `username` cookie (1-year max-age, `httponly`) is set and matched against the `users` table on every request. No passwords are stored. Content records carry a nullable `user_id` FK so the library is filtered per user.

## Data Model

| Table | Description |
|---|---|
| `users` | Usernames; one-to-many with `contents` |
| `contents` | Ingested items; tracks status (`pending`, `ready`, `error`) and content type |
| `content_text` | Cleaned text, raw and reconciled transcripts, OCR corpus |
| `question_sets` | Groups of generated questions; multiple sets may exist per content (latest is active) |
| `questions` | Individual questions with `bloom_level`, `prompt`, `expected_answer`, `key_points_json` |
| `quiz_attempts` | Records of quiz sessions with `comfort_rating`, `total_score`, and submission timestamp |
| `schedule_states` | Per-content spaced-repetition state: `step_index`, `next_due_at`, `is_terminated` |
| `notifications` | Notification log with kind (`system`, `email`) and delivery status |
| `concepts` | Extracted knowledge concepts (canonical name, aliases, summary) |
| `concept_schedules` | Spaced-repetition state for individual concepts |
| `concept_evidence` | Grounded text spans linking concepts to content segments |
| `content_segments` | Fixed text chunks used as evidence anchors |
| `concept_merge_audits` | Audit log of concept deduplication decisions |

Schema migrations for existing databases are handled by `ensure_schema_compatibility()` in `app/db/engine.py`, which adds missing columns via `ALTER TABLE` without requiring Alembic.
